## Why

`quaid serve` couples the background runtime (vault-sync supervisor + extraction worker + janitor + idle-close + quarantine sweep + full-hash audit + RCRT + embedding drain) to the stdio MCP transport: when the MCP client closes stdin, `_service.waiting()` returns, the process exits, and the workers die with it. The Phase 5 design (`docs/superpowers/specs/2026-05-03-phase5-conversation-memory-design.md`) explicitly promised *"extraction is background, debounced, and survives daemon restart,"* but in practice extraction only survives client *connection*. The `daemon-install` stub (`openspec/changes/daemon-install/`) would have generated launchd/systemd units that wrap `quaid serve`, but launchd routes stdio to `/dev/null` — the wrapped process would EOF immediately and loop. Issues `#175` (multi-agent HTTP transport) and `#177` (standalone extraction worker) describe two symptoms of the same architectural gap. This proposal closes both and supersedes `daemon-install`.

## What Changes

- Introduce a new `quaid daemon run` foreground entry point that owns the **full background runtime** (vault-sync supervisor, watchers, extraction worker, janitor, idle-close, quarantine sweep, full-hash audit, RCRT, embedding drain) and **never opens stdio MCP**. With `--http [--port N] [--bind addr] [--token-file PATH]`, the daemon MAY additionally host an MCP HTTP/SSE transport in-process — there is no separate "transport sidecar" service. This is what launchd/systemd execute.
- Introduce a new **`serve_host` session type** for the no-daemon fallback. The session-type set becomes `{ daemon, serve_host, serve, cli }`. Exactly one process per database holds runtime ownership: a live `daemon` wins; otherwise the first `serve` that atomically claims the runtime-host lease promotes itself to `serve_host`. Plain `serve` and `cli` are transport-only / one-shot. Two concurrent `quaid serve` processes against the same DB with no daemon registered now produces one `serve_host` and one transport-only `serve`, not two competing runtimes.
- Keep `quaid serve` stdio-first and backwards-compatible at the surface: today's single-`serve` invocation still runs the full runtime in-process — it just does so via the `serve_host` promotion path internally rather than unconditionally.
- Add opt-in HTTP/SSE MCP transport: `quaid serve --http [--port 3112] [--bind 127.0.0.1] [--token-file PATH] [--require-loopback-token]`. Loopback (`127.0.0.1`/`::1`) is unauthenticated **only when `daemon.http.trusted_loopback = true` and no token file is configured**; with `daemon.http.trusted_loopback = false` (the new default) loopback requires a token, addressing the SSH-forward / WSL / devcontainer / shared-host threat surface. Non-loopback bind **always** requires `--token-file` and fails closed at startup if missing. Enables the `transport-sse-server` feature on `rmcp` (no new top-level dependency).
- Add `quaid daemon` CLI subcommand group: `install`, `uninstall`, `start`, `stop`, `restart`, `status`, `logs`. `install` writes a launchd plist (macOS) or systemd unit (Linux) that invokes `quaid daemon run` against the active database, with optional `--http`/`--port`/`--bind`/`--token-file` flags passed through to the daemon's argv — so the installed daemon directly hosts the SSE transport when those flags are supplied. `install` is idempotent: rerunning rewrites the unit and reloads the service.
- Add `quaid status` (new top-level command, distinct from existing `quaid stats`): reports session-type registration state, owning PID, database path, last extraction-queue activity, last vault-sync heartbeat, configured MCP transports, HTTP port/bind/auth state, and `trusted_loopback` setting when HTTP is enabled.
- `quaid daemon logs` reads the **launchd `StandardOutPath`/`StandardErrorPath`** file (macOS) and `journalctl --user -u quaid-daemon` (Linux). No `os_log` subsystem rewrite; the daemon continues to write to stdout/stderr as today and the unit captures those streams to a known file path.
- Extend the existing `serve_sessions.session_type` column (added at `src/core/db.rs:516-532` via the additive `ensure_serve_session_columns` shim, current accepted value `'serve'`) to accept `{ 'daemon', 'serve_host', 'serve', 'cli' }`. **No schema-version bump.** The expansion follows the established additive-`ALTER` pattern; existing rows default to `'serve'` and continue to behave correctly. Old binaries reading a DB with new session-type values will treat any non-`'serve'` row as a non-owner (the ownership predicates already filter by `session_type = 'serve'`), which is the safe fallback during partial-rollback windows.
- Audit and update **every existing call site** in `src/core/vault_sync/ownership.rs` that filters by `session_type = 'serve'` so it accepts `{ 'daemon', 'serve_host', 'serve' }` as valid live owners. Rename `ServeOwnsCollectionError` to `RuntimeOwnsCollectionError` (or keep the alias) so error messages no longer falsely imply "a `quaid serve` process" when the holder is actually a `daemon` or `serve_host`. Existing `serve_sessions` consumers outside `ownership.rs` are also audited; mismatched filters are listed explicitly in `tasks.md` rather than left to chance.
- Supersede the existing `daemon-install` change (`openspec/changes/daemon-install/proposal.md`); its single-paragraph stub is replaced by this proposal's daemon-lifecycle capability and can be removed when this change archives.

## Capabilities

### New Capabilities

- `daemon-runtime`: The detached foreground daemon entry point (`quaid daemon run`), the full background-duty owner map (see `design.md`), runtime-host coordination via the existing session registry (`daemon` and `serve_host` session types), optional in-process MCP/SSE hosting via `--http` flags, signal handling for graceful shutdown, and the contract that the daemon never opens stdio MCP.
- `daemon-lifecycle-cli`: The `quaid daemon install | uninstall | start | stop | restart | status | logs` surface, the launchd plist generator (macOS), the systemd unit generator (Linux), idempotent install/reload, the launchd-stderr-file-tail strategy for `daemon logs` on macOS, and the `quaid status` top-level command.
- `mcp-http-transport`: The opt-in HTTP/SSE MCP transport hosted by either `quaid serve --http` or `quaid daemon run --http`, default loopback binding, `trusted_loopback` config gate, fail-closed handling of non-loopback `--bind` without a token, fail-closed handling of loopback without a token when `trusted_loopback = false`, bearer-token file format with `0600` enforcement, SIGHUP token reload, and the `transport-sse-server` rmcp feature flip.

### Modified Capabilities

- `extraction-worker`: Adds the runtime-host contract — the worker SHALL run inside exactly one process per database at a time, identified by either a live `daemon` session or, in the no-daemon fallback, the unique `serve_host` session. `serve` and `cli` SHALL never spawn the worker. Adds the cooperative-shutdown requirement scoped at **job** granularity (not window granularity), aligning with the current extractor's per-job cursor/queue-done commit model.
- `vault-sync`: Adds the `session_type` expansion to `{ daemon, serve_host, serve, cli }`, the atomic runtime-host lease for the no-daemon fallback path, the role-precedence rule (a live `daemon` wins; otherwise the first `serve` to claim the host lease becomes `serve_host`), and the explicit modification of the `Live-serve coordination for restore/remap` requirement so `live_collection_owner` and `ensure_no_live_serve_owner` treat `daemon` and `serve_host` as live owners.

## Impact

- **Code**:
  - `src/commands/daemon.rs` (new): `install`, `uninstall`, `start`, `stop`, `restart`, `status`, `logs`, `run` subcommand handlers.
  - `src/commands/status.rs` (new): top-level `quaid status` implementation distinct from `stats`.
  - `src/commands/serve.rs`: probe `serve_sessions` for an active `daemon` session; if absent, attempt `serve_host` promotion via the new atomic lease; spawn the full runtime in whichever process holds runtime-host; otherwise run transport-only.
  - `src/core/vault_sync/session.rs`: extend `register_session` to accept a `SessionType` enum (`Daemon`, `ServeHost`, `Serve`, `Cli`); persist the value into `serve_sessions.session_type`; add `find_active_daemon_session(conn)` and `find_active_runtime_host(conn)`.
  - `src/core/vault_sync/ownership.rs`: change every `WHERE s.session_type = 'serve'` clause to accept `IN ('daemon','serve_host','serve')`; audit every call site (`live_collection_owner`, `live_collection_owner_for_root_path`, `ensure_no_live_serve_owner`, `ensure_no_live_serve_owner_for_root_path`, `acquire_owner_lease`). Rename `ServeOwnsCollectionError` to `RuntimeOwnsCollectionError` (keep a thin alias if other modules depend on the old name).
  - `src/core/vault_sync/mod.rs`: split `start_serve_runtime` into `start_daemon_runtime` (full runtime + optional HTTP), `try_promote_to_serve_host` (atomic lease claim), and `start_serve_transport` (transport-only). The existing `start_serve_runtime` becomes a thin dispatch wrapper preserving today's call signature.
  - `src/mcp/server.rs`: factor the current `run(conn)` into a transport-agnostic core plus two thin entry points: `run_stdio(conn)` (today's behavior) and `run_http(conn, HttpConfig)` (new SSE transport). Both reuse the same `QuaidServer` and tool registry.
  - `src/platform/launchd.rs` (new, `#[cfg(target_os = "macos")]`): plist generator (with `StandardOutPath`/`StandardErrorPath` pointing at `~/Library/Logs/quaid-daemon.{out,err}.log`) + `launchctl bootstrap/bootout/kickstart` helpers.
  - `src/platform/systemd.rs` (new, `#[cfg(target_os = "linux")]`): unit generator + `systemctl --user enable/start/stop/status` helpers.
  - `src/main.rs`: new `Daemon { action: DaemonAction }` and `Status` variants on the `Commands` enum.
  - `src/core/db.rs`: extend the existing `ensure_serve_session_columns` shim so the CHECK on `session_type` (or the seeded default value scheme) accepts `{ 'daemon', 'serve_host', 'serve', 'cli' }`. **No** `SCHEMA_VERSION` bump.
  - `Cargo.toml`: `rmcp = { version = "0.1", features = ["transport-sse-server"] }`.
- **MCP transport surface**:
  - stdio remains the default for `quaid serve`.
  - `quaid serve --http` opens SSE on `127.0.0.1:3112` by default (with `trusted_loopback=false`, that requires a token).
  - `quaid daemon run --http` opens SSE in the daemon process with the same flag surface.
  - All existing MCP tools work over either transport unchanged.
- **Config**: New keys in the existing mutable `config` table — `daemon.http.enabled` (default `false`), `daemon.http.port` (default `3112`), `daemon.http.bind` (default `127.0.0.1`), `daemon.http.token_file` (default `null`), **`daemon.http.trusted_loopback` (default `false`)**. CLI flags on `quaid daemon install`/`quaid serve --http` always win over config defaults.
- **Schema**: Extend `serve_sessions.session_type` via additive `ALTER` shim only; no `SCHEMA_VERSION` bump. The default for new rows in old binaries continues to be `'serve'`; new binaries pick the appropriate value (`'daemon'`, `'serve_host'`, `'serve'`, `'cli'`) at registration time. Ownership predicates are updated in lockstep so a daemon row is visible as an owner.
- **Migration**: None. Additive column-value expansion only. Mixed-binary scenarios (an older binary running alongside a new one) are safe because the older binary's ownership predicates already ignore non-`'serve'` session types — it cannot see a `daemon` or `serve_host` as an owner, which is the correct fallback behavior during partial-rollback windows.
- **Tests**:
  - `tests/daemon_runtime.rs`: `quaid daemon run` starts the full runtime, processes a queued extraction job after the foreground caller would have disconnected, shuts down cleanly on `SIGTERM`, and finishes its **current job** (not just current window) before exit.
  - `tests/daemon_serve_coordination.rs`: with a daemon registered, `quaid serve` does not spawn a runtime; without a daemon, `quaid serve` promotes itself to `serve_host` and spawns; **two concurrent `quaid serve` invocations with no daemon** produce one `serve_host` and one transport-only `serve`, never two competing runtimes.
  - `tests/ownership_session_type_audit.rs`: every code path in `src/core/vault_sync/ownership.rs` accepts `daemon` and `serve_host` as live owners; verified by inserting fixture rows of each session type and asserting the predicate result.
  - `tests/mcp_http_transport.rs`: loopback with `trusted_loopback=true` accepts unauthenticated tool calls; loopback with `trusted_loopback=false` and no token fails to start; loopback with token enforces bearer; `--bind 0.0.0.0` without `--token-file` fails to start.
  - `tests/daemon_lifecycle_cli.rs`: `install` generates a syntactically-valid plist/unit; `uninstall` removes it; `status` exit codes are correct; `install` is idempotent across reruns; `install --http` produces a unit whose `ExecStart`/`ProgramArguments` carries the HTTP flags.
  - `tests/daemon_logs.rs`: on macOS, `quaid daemon logs` tails `~/Library/Logs/quaid-daemon.err.log`; on Linux, it invokes `journalctl --user -u quaid-daemon`.
  - `tests/status_command.rs`: `quaid status` reflects session-type state, MCP transports, last extraction timestamp.
- **Dependencies**: No new top-level crates. `rmcp`'s `transport-sse-server` feature is the only Cargo flip.
- **Performance**: HTTP/SSE adds <5 ms p99 over stdio for a single MCP tool call. The daemon process baseline memory matches today's `quaid serve` (it owns the same workers).
- **Security**: HTTP transport defaults to loopback. `trusted_loopback = false` (new default) requires authentication even on loopback, closing the SSH-forward / WSL / devcontainer / shared-host gap. Non-loopback binds always require a token file (perm `0600` enforced on read, decoded length ≥32 bytes, single-line content). Token is a single line, ≥32 random bytes encoded as base64-url; rotation is "edit file + SIGHUP" or `quaid daemon restart`.
- **Docs**: `docs/operator-guide-daemon.md` (new): install, uninstall, log inspection, HTTP bind/token setup (including `trusted_loopback` guidance), troubleshooting matrix.
